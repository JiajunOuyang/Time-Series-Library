风电功率预测是电力系统运行的关键环节。我针对 WindPower 数据集完成了**超短期（24h）到中长期（180h）**的全尺度预测任务。理解到：
- 风电功率预测直接服务于**日前电力市场竞价**（24h 预测）和**机组组合优化**（72h+ 预测）
- 风电出力波动性极大（CV=0.93），是时序预测中最难的场景之一

#### 算法开发与测试流程

```
数据探索 → 特征分析 → 模型适配 → 批量训练 → 异常处理 → 结果评估

 列名兼容   相关性筛选   4个模型bug  96次自动化  6类异常   12模型×4长度
```

#### 3.1 工程化自动化：批量实验调度系统

**问题**：96 次实验如果手动逐个运行，需要持续盯屏 10+ 小时，且出错后无法自动恢复。

**我的工程方案**：

```python
# batch_runner.py — 我构建的批量实验调度系统核心架构
class ExperimentScheduler:
    """
    功能模块:
    1. 参数模板化 — COMMON_ARGS + MODEL_SPECIFIC 分层管理
    2. 自动跳过 — check_already_done() 检测已完成checkpoint
    3. 异常容错 — 6类异常自动分类 (OK/FAIL/TIMEOUT/SKIP/SKIP_SLOW/ERROR)
    4. 断点续跑 — 重启时自动跳过已完成的checkpoint
    5. 日志持久化 — 每次训练的stdout/stderr全量保存
    6. 超时保护 — 单实验900s超时自动终止
    """

# 实际运行效果
# 首次运行: 96个实验中 45个完整通过
# 自动重跑: 16个失败/超时实验自动恢复
# 最终产出: 72个有效checkpoint + 完整指标记录
```

**这个调度系统的工程价值**：
- 可以在无人值守下 24h 运行
- 支持随时中断和恢复（断点续跑）
- 所有异常自动记录分类，事后可追溯

#### 3.2 模型部署适配：跨硬件兼容性验证

**实际发现**：FFT 类模型（TimesNet、FEDformer、Autoformer）在消费级 GPU（RTX 3050）上不可用。

**工程处理流程**：
```
            发现异常 → 性能分析 → 根因定位 → 决策与处理
               ↓          ↓          ↓           ↓
          TimesNet    torch.profiler  cuFFT不支持  标记为硬件不兼容
          4.87s/iter   定位到FFT算子   消费级GPU    实验文档记录
```

**对部署的启示**：
- 模型选型前必须验证目标硬件的算子支持
- 需要建立**模型-硬件兼容性矩阵**
- 推理部署时应避免 FFT 类模型，优先选用线性/注意力模型

#### 3.3 推理优化：测试阶段的性能分析

**实际数据**：NS_Transformer 在 pred_len=720 时测试阶段就超时。

```python
# 推理复杂度分析
# 测试集: 20,336 个滑动窗口 × 每个窗口 [96+720=816] 长度序列
# encoder-decoder自注意力: O(L²) = O(816²) ≈ 666k
# 对比 pred_len=96: O(144²) = 21k
# 复杂度增加: 31x

# 优化措施
# 1. 训练阶段使用 batch_size=32，测试降至 batch_size=16
# 2. 实施训练/测试分离的时间预算 (训练900s, 测试600s)
# 3. 支持仅测试模式 (--is_training 0) 用于结果恢复
```

**推理优化的工程思考**：
- 生产环境需评估模型的**推理延迟**和**吞吐量**
- DLinear 类线性模型不仅训练快，推理也只需 O(L) 复杂度——是部署首选
- 自研模型（MuST/ProSTA）在推理阶段需关注插值和时序预测算子的计算开销

#### 3.4 系统集成：多组件兼容性修复

在集成 Time-Series-Library 框架与风电数据集的过程中，我发现并修复了 **6 个底层兼容性问题**：

| 问题 | 影响范围 | 修复方案 | 工程价值 |
|------|------|------|------|
| **硬编码路径** `./checkpoints/` | test-only 恢复全失败 | 改为 `self.args.checkpoints` | 支持自定义checkpoint路径 |
| **频率映射缺失** `15min` | TimeMixer/TiDE/TFT 崩溃 | 三处 freq_map 添加 15min 支持 | 支持15min/30min分辨率 |
| **Windows 260字符路径** | NS_Transformer 全崩溃 | 模型ID缩写 + 短checkpoint路径 | 适配Windows生产环境 |
| **GBK编码错误** | emoji打印崩溃 | 移除emoji + UTF-8环境变量 | 跨平台兼容 |
| **MuST维度bug** | pred_len=336/720 报错 | scale_factor → size=self.pred_len | 修复自研模型长预测能力 |
| **主实验短预测NaN** | 25特征+pred≤192 | 标记NaN + 留待排查 | 识别数值稳定性边界 |

**这 6 个修复的工程意义**：它们都是"从论文代码到可运行工程"的典型障碍。能够在**不熟悉代码库**的情况下快速定位并修复这些问题，体现了工程落地能力。

## 三、面试高频问题预答

### Q1: 项目中遇到的最大技术挑战是什么？

**答**：最大挑战是将 12 个不同架构的模型在同一个数据管道上稳定运行并产出可对比结果。

具体有三个难点：
1. **底层兼容性**：框架本身对风电数据格式（15min频率、time列名）不兼容，需要修改 3 个文件的数据加载和嵌入层
2. **硬件适配**：3 个 FFT 类模型在消费级 GPU 上存在 1600x 性能差异，需要建立模型-硬件兼容性矩阵
3. **工程可靠性**：构建了自动调度系统，支持断点续跑、异常分类、超时保护，确保 96 次实验无需人工盯屏

### Q2: 你是怎么保证不同模型公平对比的？

**答**：通过四层控制：
1. **参数层**：统一 seq_len、d_model、batch_size、lr、epochs
2. **数据层**：固定随机种子、固定数据划分、固定标准化方式
3. **评估层**：统一 MSE/MAE 指标、统一结果存储格式
4. **特殊参数**：仅当模型必需时才加（如 TimeMixer 的 down_sampling），并记录在文档中

### Q3: 如果要部署到生产环境，你会怎么做？

**答**：基于这次实战经验，我的部署方案是：
1. **模型选型**：优先 DLinear（训练快、推理快、精度高、无特殊依赖）作为主力模型
2. **推理优化**：batch推理 + ONNX导出 + FP16量化
3. **可靠性**：checkpoint管理 + 输入数据校验（防止频率不一致/列名不匹配/异常值） + 输出范围监控
4. **硬件适配**：提前验证目标环境算子支持，避开 FFT/大矩阵乘法等瓶颈操作
5. **降级策略**：复杂模型不可用时回退到 DLinear baseline

### Q4: 你从这次项目中学到了什么？

**答**：最大的收获是理解了**"从论文代码到工程落地"**的差距。具体来说：
1. 框架代码存在很多**隐式假设**（时间列名、频率编码、路径格式），实际数据和环境往往不满足
2. 模型跑通 ≠ 模型可用，需要验证在目标硬件上的实际性能
3. 工程化不只是写代码，还包括**异常处理、日志管理、结果组织、文档编写**
4. 批量实验的自动化调度能节省大量时间并提高可靠性

### Q5: 你怎么看待 AI 在电力行业的应用前景？

**答**：电力行业是 AI 时序预测最有价值的场景之一。具体来说：
1. **发-输-变-配-用**每个环节都产生海量时序数据，且都有预测需求
2. 新能源（风电/光伏）的高波动性使得**准确预测直接影响电网调度安全和经济性**
3. 从技术趋势看，多源数据融合（气象+SCADA+市场）、概率预测（量化不确定性）、边端推理（低延迟）是三个关键方向
4. 南瑞作为电力自动化龙头，在调度自动化、变电站自动化等领域有天然的数据和场景优势

## 四、面试时的项目讲述框架（STAR 法）

**S - Situation（背景）**
> 风电功率预测是电力系统日前调度和实时平衡的关键环节。我在一个包含 102,157 条 15 分钟分辨率、覆盖 57 台风机的真实风电场数据集上，使用 12 个 SOTA 时序预测模型（含 2 个自研模型）进行了系统性实验。

**T - Task（任务）**
> 目标是在 4 个预测长度（24h~180h）和 2 组特征方案（5特征基线 + 25特征主实验）下，完成共计 96 次训练+测试实验，产出可对比的预测结果和可视化报告。

**A - Action（行动）**
> 1. 首先进行了数据质量分析和特征工程（相关性筛选、频率编码兼容、列名适配）
> 2. 构建了自动化实验调度系统（参数模板化、断点续跑、异常自动分类、超时保护）
> 3. 在集成过程中发现并修复了框架的 6 个底层兼容性问题（硬编码路径、GBK编码、Windows路径限制等）
> 4. 识别出 3 个 FFT 类模型在消费级 GPU 上的硬件瓶颈并调整实验计划
> 5. 对 2 个自研模型进行了跨预测长度的正确性验证和 bug 修复

**R - Result（结果）**
> 1. 最终产出 72 个有效 checkpoint，覆盖 9 个模型 × 4 预测长度 × 2 组实验
> 2. DLinear 以 MSE=0.309 获得最佳性能，比 naive 预测降低 58% 误差
> 3. 自研模型 MuST 达到 MSE=0.320，与 SOTA 模型性能持平
> 4. 产出了完整的预测曲线图、指标汇总表、对比报告和技术文档
