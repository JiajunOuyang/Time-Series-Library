# 风电预测模型训练问题与应对指南

> 基于 Time-Series-Library 在 WindPower 数据集上 12 个模型、96 次实验的实战经验总结。

---

## 1. 训练前数据分析

### 1.1 风电数据特有的预处理

#### 时间列命名兼容性

**问题描述**：TSLib 的 `Dataset_Custom` 在第 270 行硬编码了 `cols.remove('date')`，要求时间列必须命名 `date`。风电数据常见的时间列名为 `time` 或 `timestamp`，直接使用会触发 `ValueError: 'date' is not in list`。

**应对策略**：
```python
# 方案1：重命名列（简单直接）
df.rename(columns={'time': 'date'}, inplace=True)

# 方案2：创建自定义Dataset类（适合需要保留原始列名时）
class Dataset_WindPower(Dataset_Custom):
    def __read_data__(self):
        # ... 自定义读取逻辑，将time映射为date
```

#### 频率编码缺失

**问题描述**：风电数据常用 15min / 30min / 10min 等频率，但 TSLib 的 `layers/Embed.py` 第 100 行 `freq_map` 仅支持 `h/t/s/m/a/w/d/b`，使用 `15min` 会触发 `KeyError: '15min'`，导致 TimeMixer、TiDE 等使用时间嵌入的模型崩溃。

**影响范围**：`layers/Embed.py`、`models/TiDE.py`、`models/TemporalFusionTransformer.py` 三处均需修改。

**应对策略**：
```python
# layers/Embed.py 第100行，追加高分辨率频率
freq_map = {'h': 4, 't': 5, 's': 6,
            'm': 1, 'a': 1, 'w': 2, 'd': 3, 'b': 3,
            '15min': 5, '30min': 5, '10min': 5, '4h': 4, '2h': 4}
# 15min与't'(minutely)同为5个时间特征：月、日、星期、时、分
```

#### 特征与目标相关性分析

**问题描述**：WindPower 数据集包含 291 列（1个time + 5个聚合特征 + 285个单风机特征），但并非所有特征都有预测价值。**不经筛选直接全量输入会导致维度灾难和噪声干扰。**

**实际数据分析结果**：

| 特征 | 与YG相关系数 | 预测价值 |
|------|:----------:|:------:|
| FS（风速） | **0.8672** | ✅ 核心特征 |
| ZT（转速） | 0.3046 | ⚠️ 中等 |
| WG（无功功率） | 0.0937 | ❌ 弱相关 |
| FXJ（风向角） | 0.0133 | ❌ 几乎无关 |
| 单风机YG | 0.92~0.96 | ✅ 极强（但接近标签泄漏） |

**应对策略**：
```python
# 计算所有特征与目标的Pearson相关系数
corrs = df.iloc[:, 1:].corrwith(df['YG']).abs().sort_values(ascending=False)
# 选取Top-K + 保留核心物理特征
selected = corrs.head(20).index.tolist()
final_features = ['YG', 'WG', 'FS', 'FXJ', 'ZT'] + selected
```

> ⚠️ **注意**：单风机 YG 与总 YG 相关性达 0.93-0.96（总 YG ≈ Σ单风机YG），作为特征时模型几乎退化为"求和器"。**建议排除单风机 YG**，改用相关性高的非 YG 特征（如各风机 FS、ZT）。

### 1.2 数据质量报告实例


### 💡 经验教训

1. **永远先检查列名和频率兼容性**——我们在 `date` 和 `15min` 上卡了 2 小时，都是 5 分钟就能修的问题。
2. **相关性分析比盲目加特征重要**——基线 5 特征中有 3 个几乎没用，主实验 25 特征中 20 个是单风机 YG（接近标签泄漏），实际有效增量有限。
3. **特征工程先于模型调参**——在这个数据集上，换特征的收益远大于换模型。

## 2. 长时预测 vs 短时预测（风电场景）

### 2.1 风电场景下的定义

| 类型 | 预测时长 | 数据对应 | 应用场景 |
|------|:------:|:------:|------|
| 超短期 | 0-4h | pred_len ≤ 16 | 实时调频、风机控制 |
| 短期 | 4-24h | pred_len 16-96 | 日前电力市场竞价 |
| 中长期 | 24h+ | pred_len 96-720 | 机组组合、检修计划 |

### 2.2 误差累积分析

**实际实验结果**（以 DLinear 为例，标准化 MSE）：

```
pred_len=96  (24h):   MSE=0.309 → RMSE_orig=931  → 误差/均值=51.6%
pred_len=192 (48h):   MSE=0.415 → RMSE_orig=1079 → 误差/均值=59.8%
pred_len=336 (84h):   MSE=0.520 → RMSE_orig=1208 → 误差/均值=67.0%
pred_len=720 (180h):  MSE=0.654 → RMSE_orig=1354 → 误差/均值=75.1%
```

**规律**：预测长度翻倍 → MSE 增长约 40-60%。从 24h 到 180h，RMSE 增加了 45%。

### 2.3 模型在不同预测长度上的表现差异

| 模型 | 96 (24h) | 720 (180h) | 短期→长期退化 |
|------|:------:|:---------:|:----------:|
| DLinear | 0.309 | 0.654 | +112% |
| PatchTST | 0.316 | 0.759 | +140% |
| NS_Transformer | 0.448 | 0.924 | +106% |
| Informer | 0.387 | 0.985 | +155% |

- **DLinear 在短时和长时都最稳定**，退化幅度最小
- **Informer 长时退化严重**（ProbSparse 注意力在风电高噪声场景下稀疏度不稳定）
- **Transformer 类模型在长预测时测试阶段经常超时**（encoder-decoder 架构在长序列上计算量 O(n²)）

### 2.4 超时问题：长预测的特殊挑战

**实际现象**：NS_Transformer 在 pred_len=336/720 时，即使已收敛（checkpoint 保存），测试阶段仍超出 900s 限制。

**原因**：
- Decoder 需要处理 `label_len + pred_len = 48 + 336 = 384` 长度的序列
- 自注意力复杂度 O(L²) = O(384²) = O(147k)，远大于 pred_len=96 的 O(144²)=O(21k)
- 测试集有 ~20k 个滑动窗口，每个都需完整前向传播

**应对策略**：
```python
# 1. 分离训练和测试超时
TIMEOUT_TRAIN = 900   # 训练 15min
TIMEOUT_TEST = 600    # 测试 10min (测试通常更快)

# 2. 对encoder-decoder模型减小batch_size
if model in ['Nonstationary_Transformer', 'Informer']:
    test_batch_size = 16  # 低于训练batch_size=32

# 3. 允许checkpoint已存时跳过，仅跑test-only恢复
python run.py --is_training 0 --model_id WP_B_NT_336 ...
```

### 💡 经验教训

1. **短预测时各模型差异不大**（MSE 0.309~0.448），选最简单的 DLinear 即可。**长预测时模型选择才有关键影响**。
2. 风电纯数据驱动预测超过 72h 基本不可靠（RMSE > 均值 65%），**实际应用需要 NWP（数值天气预报）作为外源输入**。
3. **测试阶段的耗时常被低估**——encoder-decoder 模型在 pred_len=720 时测试比训练还慢。

## 3. 训练参数设置与调优

### 3.1 基于风电数据特点的参数建议

| 参数 | 推荐值 | 理由 |
|------|:------:|------|
| `d_model` | 256 | 风电数据维度高但信噪比低，过大易过拟合噪声 |
| `d_ff` | 512 | 与 d_model 保持 2:1 比例 |
| `batch_size` | 32 | 数据噪声大，较大 batch 稳定梯度 |
| `learning_rate` | 1e-4 | 风电非线性强，大 lr 容易不收敛 |
| `dropout` | 0.1 | 适当正则化应对噪声 |
| `patience` | 3 | 风电训练波动大，短的 patience 可防止在噪声中过度训练 |
| `factor` | 3 | Auto-correlation/Autoformer 相关模型的注意力因子 |

### 3.2 多模型公平比较的保证措施

```python
# 统一配置（所有模型一致）
COMMON_ARGS = {
    'seq_len': 96,          # 统一输入长度
    'label_len': 48,        # 统一解码前缀
    'd_model': 256,         # 统一隐层维度
    'd_ff': 512,            # 统一FFN维度
    'batch_size': 32,       # 统一批次大小
    'train_epochs': 10,     # 统一最大epoch
    'patience': 3,          # 统一早停标准
    'learning_rate': 1e-4,  # 统一学习率
    'num_workers': 0,       # 统一为0避免多进程差异
    'seed': 2021,           # 固定全局随机种子
}

# 模型特殊参数（仅当必需时才加）
MODEL_SPECIFIC = {
    'TimeMixer': {'down_sampling_layers': 1, 'channel_independence': 0},
    'MuST': {'scales': [96, 48, 24, 12, 6]},
}
```

### 3.3 FFT 模型的参数调优实例

**问题**：TimesNet 在 RTX 3050 上 `d_model=256` 时每步 4.87s，预测单次训练需 ~30 小时。

**调优路径**：

| 迭代 | 配置 | 速度 | 结果 |
|:--:|------|:--:|------|
| 初始 | d_model=256, d_ff=512, batch=32, top_k=5 | **4.87s/iter** | ❌ 不可用 |
| 优化1 | d_model=128, d_ff=256, batch=16 | ~1.0s/iter | ⚠️ 仍超时 |
| 优化2 | + top_k=3, epochs=5 | ~0.8s/iter | ⚠️ 仍超时 |
| 结论 | **FFT 在消费级 GPU 上是硬件瓶颈** | — | ❌ 放弃 |

**根本原因**：`torch.fft.rfft` 在 RTX 3050 上没有 cuFFT 加速。**数据中心 GPU（A100/H100/V100）才有**。

### 💡 经验教训

1. **不要假设所有模型在同一硬件上可运行**——FFT 类模型（TimesNet、FEDformer、Autoformer）在消费级 GPU 上根本不可用。
2. **batch_size 和 num_workers 在 Windows 上需特别注意**——multiprocessing 在 Windows 上有 spawn 限制，`num_workers=0` 是安全默认值。
3. **统一参数 ≠ 公平比较**——有的模型需要额外参数才能运行（如 TimeMixer 的 `down_sampling_layers`、MuST 的 `scales`），不提供这些参数会导致模型崩溃而非性能差异。

## 4. Epoch 与训练步长设置

### 4.1 风电数据的 Epoch 计算

```
总样本:     102,157 条
训练集:     71,318 条 (70%)
验证集:     10,122 条 (10%)
测试集:     20,336 条 (20%)

seq_len=96, batch_size=32:
  Train steps/epoch = (71318 - 96 - 96 + 1) / 32 ≈ 2,229
  Val steps/epoch   = (10122 - 96 - 96 + 1) / 32 ≈ 310
```

**实际训练数据量参考**：

| 数据规模 | 样本量 | Steps/Epoch | 建议Epoch |
|------|:------:|:---------:|:--------:|
| 单风机3个月 | ~13,000 | ~420 | 20-50 |
| 单风机1年 | ~52,500 | ~1,700 | 10-20 |
| 多风场2年 | ~100,000+ | ~2,200+ | 5-10 |

### 4.2 Early Stopping 实践

**实际观察**：10 个 epoch 的训练中，绝大多数模型在第 3-5 个 epoch 触发早停：

| 模型 | 实际停止Epoch | MSE（96） |
|------|:----------:|:------:|
| DLinear | 3 | 0.309 |
| iTransformer | 5 | 0.326 |
| TimeMixer | 5 | 0.339 |
| PatchTST | 4 | 0.316 |
| NS_Transformer | 10 (未早停) | 0.448 |

**建议**：
- `patience=3` 对 10 epoch 的短训练足够
- 对 `train_epochs=50+` 的长训练，`patience=5-10`
- **验证集 MSE** 是更好的 monitor 指标（比 loss 更直接反映预测质量）
- 风电数据噪声大 → 验证 loss 抖动剧烈 → patience 不宜设太小（会过早停止）

### 4.3 两个配置实例

**小规模（单风机 3 个月）**：
```bash
python run.py \
  --train_epochs 30 \
  --patience 10 \
  --batch_size 64 \
  --seq_len 96 \
  --pred_len 24 \
  --d_model 128
# 预计训练时间: 10-20 min
```

**大规模（多风场 2 年）**：
```bash
python run.py \
  --train_epochs 10 \
  --patience 3 \
  --batch_size 32 \
  --seq_len 96 \
  --pred_len 96 \
  --d_model 256
# 预计训练时间: 2-8 min (取决于模型复杂度)
```

### 💡 经验教训

1. **WindPower 10 万样本对大多数模型已经足够**——早停在第 3-5 epoch，说明模型快速收敛到数据的信息上限。
2. **不要用训练集 loss 判断收敛**——风电数据噪声大，训练 loss 可以继续下降但验证 loss 已经回升。
3. **NS_Transformer 跑满 10 epoch 仍未早停**但最终 MSE(0.448) 比早停的 DLinear(0.309) 还差——**早停 ≠ 性能差，不早停 ≠ 性能好**。

## 5. 算力问题

### 5.1 实际遇到的瓶颈

#### FFT 算子瓶颈

**现象**：TimesNet 每步 4.87s（DLinear: 0.003s），**慢了 1600 倍**。

**排查过程**：
```python
# 1. 确认模型是否正确加载到GPU
print(next(model.parameters()).device)  # cuda:0 ✓

# 2. 检查GPU利用率
nvidia-smi  # 100% → GPU在工作，但算得慢

# 3. pdb逐层排查
import torch.profiler
# 发现 torch.fft.rfft 单次调用耗时 4.5s
```

**结论**：RTX 3050 没有 cuFFT 加速，FFT 操作退化为 CPU 执行。

**应对**：
- TimesNet、FEDformer、Autoformer → **跳过，标记为硬件不兼容**
- 其他 9 个模型在 RTX 3050 上完全可运行

#### Windows 路径长度限制

**现象**：`Nonstationary_Transformer` 因模型名太长（29 字符）导致 checkpoint 路径超过 260 字符，触发 `FileNotFoundError: [WinError 206]`。

```python
# 原始路径 (272 字符, 超出260限制):
./checkpoints/long_term_forecast_WindPower_baseline_Nonstationary_Transformer_96_Nonstationary_Transformer...

# 修复后 (238 字符):
./c/long_term_forecast_WP_B_NT_96_Nonstationary_Transformer...
```

**应对策略**：
```python
MODEL_SHORT = {
    'Nonstationary_Transformer': 'NT',
    'TimeXer': 'TX', 'iTransformer': 'iT', ...
}
model_id = f"WP_{group_short}_{model_short}_{pred_len}"  # 极简命名
# 搭配 --checkpoints ./c/ 缩短基础路径
```

### 5.2 显存管理

在 RTX 3050 (6GB/8GB VRAM) 上的实际表现：

| 配置 | 显存占用 | 状态 |
|------|:------:|:----:|
| d_model=256, batch=32, enc_in=5 | ~2.0GB | ✅ 正常 |
| d_model=256, batch=32, enc_in=25 | ~2.5GB | ✅ 正常 |
| d_model=512, batch=32, enc_in=25 | ~4.5GB | ✅ 可运行 |
| d_model=512, batch=64, enc_in=290 | 预计 >6GB | ⚠️ 需监控 |

**本实验中未触发 OOM**——最大 enc_in=25 足够保守。

### 5.3 批量实验资源规划

```
96 次实验 (12模型 × 4pred_len × 2组)
────────────────────────────
成功运行的 9 模型:
  快速模型 (DLinear, PatchTST):   ~1-2 min/run
  中等模型 (iT, TimeXer, TM):     ~3-6 min/run  
  慢速模型 (NS_T, Informer):      ~8-14 min/run
  MuST, ProSTA:                   ~3-8 min/run

跳过 3 模型 (FFT瓶颈):
  TimesNet, FEDformer, Autoformer

总耗时: 约 8-10 小时
checkpoints 总大小: ~500MB (72个)
```

### 💡 经验教训

1. **消费级 GPU 跑 FFT 模型 = 自找麻烦**——时序预测选模型前先看有没有频域操作。
2. **Windows 260 字符路径限制是真坑**——模型名一长就炸，改名一定要短（3-5 字符）。
3. **96 次实验预计 10 小时，实际因调试和重跑花了 20+ 小时**——debug 时间永远是估计的 2 倍。
4. **checkpoint 目录和 results 目录分开管理**——`checkpoints` 存模型，`results` 存指标，防止清理时误删。

## 6. 超时与其他异常

### 6.1 实际遇到的训练中断问题

#### 场景1：测试阶段硬编码路径

**现象**：`--is_training 0`（test-only 模式）总是报 `FileNotFoundError: ./checkpoints/...`，即使指定了 `--checkpoints ./c/` 也无效。

**排查**：
```python
# exp/exp_long_term_forecasting.py 第186行（原始代码）：
self.model.load_state_dict(
    torch.load(os.path.join('./checkpoints/' + setting, 'checkpoint.pth'))
)
#                     ^^^^^^^^^^^^^^^^ 硬编码了 ./checkpoints/！
```

**修复**：
```python
# 改为使用 args.checkpoints 参数：
self.model.load_state_dict(
    torch.load(os.path.join(self.args.checkpoints + setting, 'checkpoint.pth'))
)
```

#### 场景2：GBK 编码崩溃

**现象**：`exp_basic.py` 第 96 行的 emoji 打印导致 Windows GBK 终端报错。

**修复**：
```python
# 原始: print(f"🚀 Lazy Loading: {key} ...")
# 修复: print(f"[Lazy Loading] {key} ...")
# 并设置: os.environ['PYTHONIOENCODING'] = 'utf-8'
```

#### 场景3：MuST 维度不匹配

**现象**：MuST 在 pred_len=96/192 正常，但 336/720 报 `RuntimeError: The size of tensor a (336) must match the size of tensor b (288)`。

**根因**：`models/MuST.py` 第 222 行使用 `scale_factor=c_k` 做插值，导致不同 scale 输出不同长度。

**修复**：
```python
# 原始:
mu_expand = F.interpolate(..., scale_factor=c_k, mode='nearest')
# 修复: 统一输出到 pred_len
mu_expand = F.interpolate(..., size=self.pred_len, mode='linear')
```

#### 场景4：主实验短预测 NaN

**现象**：主实验（25 特征）在 pred_len=96/192 时，几乎所有模型的 metrics.npy 都是 NaN，但 pred_len=336/720 正常。

**排查**：
```python
pred = np.load('results/.../pred.npy')  # [20336, 96, 1]
true = np.load('results/.../true.npy')  # [20336, 96, 1]
print(np.isnan(pred).any())  # True ← 预测值含NaN！
print(np.isnan(true).any())  # False ← 真实值正常
```

**根因**：25 特征 + 短预测长度组合下，部分模型输出 NaN（可能是数值不稳定或梯度爆炸）。数据本身没有问题。

**应对**：标记为 NaN，不参与指标计算，留待后续排查。

```python
    try:
        result = subprocess.run(cmd, timeout=TIMEOUT, ...)
        if result.returncode == 0:
            status = 'OK'
        else:
            status = 'FAIL'  # 记录并继续
    except subprocess.TimeoutExpired:
        status = 'TIMEOUT'   # checkpoint可能已保存，后续可补测
    # 无论结果如何，继续下一个实验
```

### 💡 经验教训

1. **硬编码路径是万恶之源**——`./checkpoints/` 写死在代码里，导致所有 test-only 恢复全失败，排查了 2 小时才发现。
2. **模型自身的 bug 比环境问题更难发现**——MuST 的维度 bug 只在特定 pred_len 触发，96/192 正常、336/720 崩溃，容易被误判为"长预测太难"。
3. **Windows + Python 的编码问题永远在暗处等着你**——emoji、中文路径、反斜杠，每个都可能是坑。
4. **NaN 不一定是数据问题**——主实验 96/192 的 NaN 是模型数值稳定性问题，数据是干净的。
5. **自动化比人工盯盘靠谱**——96 次实验如果每次出错都要手动处理，人会疯。脚本自动记录+跳过+补跑是最佳实践。
